Ajax History

Ajax requests are great for a number of reasons, but standard requests leave a lot to be desired when it comes to preserving state.

Since HTTP is a stateless protocol, and Ajax requests happen over HTTP, Ajax requests are stateless too.

When serving up static content, the URL acts as a state preserver in the sense that the content located at that resource locator will be the same each time the URL is visited.

Dynamic content is a little trickier, but state-wise, content served up by a dynamic, database (or datastore) driven backend is still static content by the time it gets to the browser.

Ajax requests, however, frequently update just parts of a page in response to state data, and the standard response doesn't modify window.history (the object responsible for maintaining a cache of URL breadcrumbs), so a group of XMLHttpRequests will never remember the steps needed to recreate their execution.

A common modern approach to solving the history problem (also referred to as "the back button problem") is to use the pushState API to update the URL each time an Ajax request is made, which looks something like this:

window.history.pushState({page: someHash}, "pageTitle","#" + someHash);

The someHash variable in this example can be any String. The window.history.pushState method updates the window.history object to contain the URL defined as the 3rd parameter of the pushState method, which in this case is #hash - again, hash being some unique identifier for the resource being loaded.

This method is usually found in the success block of an Ajax request, although it can also be located in the beforeSend block, which may even be the best option since the URL will get updated even in the event that the request fails (mirroring how a browser still generates a 404 error and corresponding URL on failure).

The window.history.pushState method should also execute on page load. Several jQuery plug-ins exist to help manage the sometimes complex code needed to effectively manage history, and jQuery BBQ (back button query) is one of the most popular.

Designing a simple REST-inspired API

REST, or REpresentational State Transfer, is a model for standardizing the way that HTTP requests and responses are handled such that several key actions can be performed. While simplified, these actions, called verbs, can map 1-1 to relational database operations, and are outlined in the table below:

REST verb Database operation
GET SELECT
POST INSERT
PUT UPDATE
DELETE DELETE

In most cases, at least the GET verb will be included in an API implementation, but the POST, PUT, and DELETE verbs are optional.

A common way to build modern APIs is to use specially crafted JSON objects to send requests and receive responses, using the server to perform the translation.

For example:

if($_GET) {
    // perform query to select data
    $result = $mysqli->query("SELECT * FROM table");

    // convert data to JSON
    $obj = $result->fetch_object();
    $json = json_encode($obj);

    // return JSON
    print $json;
}

Or a GET request that passes in a specific parameter to return:

if($_GET["id"] > 0) {
    $id = $_GET["id"];
    // perform query to select data
    $result = $mysqli->query("SELECT * FROM table WHERE '$id'");

    // convert data to JSON
    $obj = $result->fetch_object();
    $json = json_encode($obj);

    // return JSON
    print $json;
}

A cURL request is used to send an API request, and the response is returned as a String, which can then be passed to a JavaScript file and used as an object, like this:

// file making cURL request
    $url = "http://example.com/api/";
  $ch = curl_init();
curl_setopt($ch, CURLOPT_HEADER, 0);
curl_setopt($ch, CURLOPT_RETURNTRANSFER, 1);
curl_setopt($ch, CURLOPT_URL, $url);
$curlResponse = curl_exec($ch);
curl_close($ch);

print $curlResponse;

If the return type (type in jQuery) is json, then the response object can be treated as a JSON object and parsed accordingly.

Passing an additional method parameter allows for the creation of a conditional tree server-side that can decide which database query to perform and resulting JSON object to return based on the input to the script.

Backbone.js

Backbone.js is a client-side JavaScript framework that allows an MVC approach to structuring and displaying data.

As pages and scripts grow, combining model data and display rules in the same scripts can lead to complicated, low performance code that is very hard to debug - especially in situations where lots of data is being passed back and forth and lots of different page sections are being updated at the same time.

A simple explanation of the structure of a Backbone application is "Views display Collections of Model data in response to DOM events." Defining Backbone this way shows that the basic concept isn't much different than using standard Ajax requests, but the conventions that are imposed by the framework help to eliminate code redundancy.

The default data transport mechanism that Backbone uses is JSON, so a normal Backbone model will define JSON properties and methods.

A model defines the prototype for one piece of data, but it's likely that multiple model objects will exist in a single page, so collections can be defined that represent storing multiple model objects.

A collection needs to be stored somewhere, so several different data storage adapters can be defined in a collection, with the most common being an interface to a database (via an API), or a locally based testing and development solution (via the browser's localStorage). It is here in the collection that the server-side of the data reflection property is handled.

Finally, a view defines a container element, an optional HTML template using the underscore.js library (or other), a render function that sets up the default state of the view with model data, and an events function that binds DOM events to methods that update the rendered view.
